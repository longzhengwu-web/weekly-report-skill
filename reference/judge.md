# 报告质量评分（judge）

用固定维度对照标准给报告打分、指出问题、给修改建议。两个用途：
1. **生成后自检自修**：写完报告先按本表自评，低分项当场改了再给用户（见 SKILL.md「Self-evolution」自查步骤）。
2. **按需评审（HIL）**：用户贴一份周报/日报让你"评一下质量如何"，就按本表打分 + 给可执行改进 + 建议修改版。

判分要诚实、可执行：指出具体扣分点和怎么改，而不是只给个分。引用证据（哪一条、缺什么）。

---

## 一、周报评分（四维，各 100，总分取四维综合）

| 维度 | 看什么 | 高分特征 / 常见扣分 |
|---|---|---|
| **1. 完整性（信息覆盖）** | 五模块是否齐全（总结/本周完成/进行中/下周重点/风险同步）；每条是否"背景→产出→（约束）→下一步"完整弧线，不是流水账 | 扣分：缺模块；某条处理过轻（无产出/无下一步）；只罗列做了啥 |
| **2. 数据支撑（量化程度）** | 关键结论是否有指标/数字 + 参照系；主线数据是否扎实 | 扣分：只有定性描述、无量化产出（如影响范围、频率、前后对比、MR 等佐证） |
| **3. 叙事清晰度（逻辑结构）** | 主次/分层是否清晰、读者能否一眼抓重点；风险与完成是否分开 | 扣分：单条信息密度过高、混了多个子任务该拆未拆；顺序混乱 |
| **4. 风险意识（主动暴露问题）** | 风险同步是否"具体风险 + 当前状态 + 下一步/应对"三要素齐；是否真在 track 不确定性 | 高分：真实暴露不确定性而非凑数。扣分：风险空泛、无应对、或漏报明显风险 |

**输出格式（周报评分）：**
```
## 周报质量评分
1. 完整性（信息覆盖）— XX/100
   <一句结论> + <小扣分：具体哪条/缺什么>
2. 数据支撑（量化程度）— XX/100
   <列出扎实的数据点> + <小扣分>
3. 叙事清晰度（逻辑结构）— XX/100
   <好在哪> + <小扣分：哪条该拆/太密>
4. 风险意识（主动暴露问题）— XX/100
   <评价> + <小扣分>

总分：XX / 100
一句话评价：<主线/数据/风险如何 + 唯一明显提升空间>
```

## 二、日报 / 单条评分（对照三段式标准）

**标准**：每条 = 做了什么 → 产出/结论 → 下一步；约 100 字以内；有信息量；**结论 > 数量**（不能只有产出数量、没有判断价值）。

**评级**：`✅✅ 质量高` / `✅ 基本合格` / `⚠️ 需改进` / `❌ 不合格`。

**逐条要查**：
- 「做了什么」是否清楚具体？
- 「产出/结论」是否有实质判断（指标/决策/状态变化），而非只报数量？
- 「下一步」是否有，且具体可执行？（最常见缺项就是没下一步）
- 有无孤零零的裸词（如末尾单独一个 `notion` / `mr`）→ 应写"详见 Notion / 已提 MR"。
- 措辞是否口语/拗口（如"修 review 多点""明日"）→ 改书面（"处理 code review 反馈若干项""下一步"）。
- 有没有丢失有价值的技术判断/依赖/局限。

**输出格式（单条评分）：**
```
评价：<评级>
做得好的：
- …
需要改进：
- <具体问题 + 怎么改>
建议修改版：
<重写后的该条，体现三段式、补齐下一步、措辞书面化>
```

---

## 三、与自查的关系
生成报告时，用「一、二」的维度做一遍内部自查：周报四维有明显短板的、日报条目缺下一步/只有数量的，**当场改了再输出**，不必把分数展示给用户（除非用户要求评分）。用户明确要"评一下/打分"时，才输出完整评分。
