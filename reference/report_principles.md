# 报告核心原则（日报 / 周报共用）

日报和周报**要求相同**，只是时间粒度不同。两者都是**给人读**的工作报告，
不是 JSON、不是流水账。

> **先读 `learned_preferences.md`**（本地自进化偏好日志）：其中的条目优先级高于本文件的默认规则；
> 与默认冲突时以其中最新的为准。用户每次纠正后把新偏好追加进去——详见 SKILL.md「Self-evolution」。

## 篇幅：极简——leader 三句话抓重点（最高优先）

leader 时间有限，要的是**最少句子里的重点**，不是长篇。硬性要求：

- **每个工作条目控制在约 3 句话**：① 做了什么 + 核心结论（含关键指标/决策）；
  ② 必要的一句补充（状态 / 数据 / 判断）；③ 下一步或风险。能两句说清就别用三句。
- **删掉所有可删的**：背景一句带过即可，不要展开推理过程、不要解释机制细节、不要排比铺陈。
  保留"结论 + 数字 + 下一步"，其余从简。
- 仍要满足"结论 > 数量"：精简是删冗余，**不是删结论**。指标、决策、风险一个都不能少，只是说得更短。
- 自检：这一条 leader 扫一眼能否抓到"做了什么、结果如何、然后呢"？能 → 够了，别再加。
- **聚合不等于堆叠**：把多天日报/多会话聚成一条周报条目时，**同样压到约 3 句**——概括整周该线程的
  进展与结论，**不要逐日罗列每个子步骤、不要把每个阶段都展开成段落**。宁可只给"起点→现状→结论+下一步"。

## 每一条都遵循三段式：做了什么 → 产出 / 结论 → 下一步（紧凑表达）

在"约 3 句话"的篇幅内覆盖：

- **做了什么**：一句说清——做了什么、（必要时）为什么。背景一笔带过，不展开。
- **产出 / 结论**（最重要）：用**最关键的指标 / 决策 / 状态变化**给出结论，带参照系（多少、占比、影响范围）。
  决策思路只在确实影响判断时补一句，不长篇推演。
- **下一步**：一句，具体、可执行、有优先级；关键结论缺失就说明卡在哪、何时能出。
  （**风格差异**：正常版每条带下一步；**简洁版**不在条目内写下一步、统一汇总到「进行中」「下周重点」——见 `styles.md`。）

按重要性排序：对业绩/目标贡献大的、决策性的、风险性的放前面，琐碎的靠后或合并。

**按"主线/事件"聚合，而不是按会话或阶段拆条**：同一件事的不同**方面或阶段**要合并成一条，
讲清来龙去脉，不要并列成两条。例如"做缓存功能/上线"和"缓存命中率实测/监控"是同一主线的
两个阶段（功能 → 指标），**必须写成一条**："背景 → 做了功能/上线 → 实测口径与结论 → 下一步"，
**不要拆成两条还都标(主线)**。判断是否该合并：它们是否服务于同一个目标/同一个项目的不同环节？
是 → 合并成一条弧线。生成后自检：有没有两条其实是同一项目的不同阶段/侧面？有 → 合并。

**跨源去重（Claude Code ⇄ Codex）**：采集器只在**同源内**按首条 prompt 合并分叉，**不同源之间不会自动合并**。
但同一件事很可能在两个 agent 里都聊过（每条线程带 `source` 字段）。因此聚合时**必须按语义跨源合并**：
若一条 Claude 线程与一条 Codex 线程指的是同一项目/同一任务（看 cwd、项目名、文件、主题），**合并成一条**，
取信息更全的一边为主、另一边补充，证据(会话数)合并计入，**不要因为来自不同 agent 就当成两件事重复列出**。
**合并后直接给最终内容**，正文里**不要出现"来源/跨源合并/Codex/Claude"之类的标注或说明**——读者只看事，不看它从哪个工具来。

## 质量底线：结论 > 数量（最重要的一条）

只报"做了多少"而不报"得出什么结论"是**不合格**的。数量本身没有判断价值。

> **反例**："Drive 蒸馏 judge 重跑，有 54 条产出。"
> —— 缺关键结论：修正后指标/一致性怎么变了？主要 badcase 是哪类？是否影响最终
> judge？只有 54 这个数字，读者无法判断这件事的意义。
>
> **正确写法**："Drive 蒸馏 judge 重跑（54 条）。**产出**：修正后一致性从 X→Y，
> badcase 集中在 \<类型\>，暂未发现影响最终 judge。**下一步**：补 \<某项\> 验证后定稿。"
>
> 若结论确实还没出，就**显式标注缺口**并把它写进"下一步"，不要用数量充数。

每写完一条，自检：去掉所有数字后，这条还说得清"得出了什么、所以呢"吗？
说不清 → 它缺结论，回去补或标为待跟进。

## 结构：公司/个人两份，每份用"总结 / 本周完成 / 进行中 / 下周重点 / 风险同步"

报告分**公司**和**个人**两份**独立**报告（混在一起难分辨）。**每份都是自包含的**，
各自拥有完整的 5 个板块——**总结、本周完成、进行中、下周重点、风险同步全部按公司/个人各写一套**，
不要做一个跨公司/个人的全局「进行中 / 下周重点 / 风险同步」。某份无内容则整份省略。每份采用如下板块：

1. **总结** —— **1 句话**点出本周主线/重心即可，**不要赘述**：不要把"本周完成"里每条都在总结里
   复述一遍、不要逐条罗列。只交代"本周以什么为主线、并行带了哪些方向"这一层，细节留给「本周完成」。
   - 赘述（反例）：把 7 条工作各用一个分句重说一遍。
   - 凝练（正确）：「本周以文件类多级缓存降本上线为主线，并行推进召回实验、蒸馏评测复核及若干线上/数据工程事项。」
2. **本周完成** —— 按**主题/项目**分小节（`### 主题`），密实写清做了什么 + 关键结论/指标，
   **按重要性排序**（主项在前）；有 MR/Notion/文档的在句末附"相关："并标状态；有评测/对比数据时用小表格。
   下一步写法随风格：**正常版**每条可带一句下一步；**简洁版**条目内不写下一步、统一归并到「进行中」「下周重点」
   （见 `styles.md`）。
3. **进行中** —— 尚未完成、正在推进的事（一句一条）。
4. **下周重点** —— 下周要做的关键事项（一句一条，按优先级）。
5. **风险同步** —— 见下面「风险同步」板块。

归类流程：

1. **先自己按 `reference/project_categories.md` 预判**公司/个人，直接呈现分好的两份报告。
2. **再主动用 AskUserQuestion 问用户**确认或调整归属。
3. 把最终归类**写回 `reference/project_categories.md`** 与记忆，让下次预判更准。

不要用 emoji/图标做标题装饰（保持简约）。

## 必须覆盖的三个面向

1. **工作内容** —— 本期实际推进的事。
2. **任务进度** —— 各任务处于什么状态（推进中 / 完成 / 阻塞 / 待确认），
   有无状态变化（如 待审 → 已批）。
3. **可能风险** —— 见下面「风险同步」。

## 人物提取（团队共享日报必做）

团队日报/周报常是**多人共享**的（每条以同事名字开头）。生成"本人"的报告时，**先只抽取归属于本人的条目**：

- **本人是谁**：默认是当前用户。姓名/别名见 `project_categories.md` 的「本人身份」（英文名 / 中文名 /
  账号别名）。用户也可在请求里临时指定要总结谁。
- **按条目归属抽取**：每条归属于**行首那个名字**。只保留本人名字开头的条目，丢掉其他同事的。
- **当心相邻误抓**：不要把上一条/下一条同事的内容算进来；也不要因为本人名字在某条里被提及（作为协作者）
  就把那条算成本人的——**只看条目的归属人（行首名字）**。
- **协作者保留为上下文**：条目里出现的其他人（"与 X 对齐""配合 Y"）是协作者，写报告时可一句带过，
  但该条仍是本人的产出。
- 抽取后，再按主题聚合、生成报告（同一主题跨天合并成进展弧线）。

## 风险同步（仅周报的独立板块）

**周报末尾必须有一个「风险同步」板块**，主动把本周工作的风险汇总同步给 leader——这是周报的硬性要求。
**日报不单设此板块**：把卡点/风险并进对应条目的"下一步"里一句带过即可。风险**不止安全**，更多是进度类。覆盖这些类型：

- **进度/阻塞**：被卡住、无法推进（如等外部资源、等他人）。
- **延期**：预计赶不上原定时间 / 结论未能按期产出。
- **依赖**：依赖其他团队、评审、外部系统才能继续。
- **待确认**：方案/结论尚未拍板，存在返工可能。
- **质量**：指标未达预期、效果不及目标（如命中率偏低）。
- **成本**：成本异常或超预算。
- **安全**：安全隐患、合规风险。

写法：每条风险讲清 **① 什么有风险 ② 原因 ③ 影响（拖慢什么/波及什么）④ 应对或需要的支持**。
- 风险既要在对应工作条目里点到，也要在末尾「风险同步」里**汇总**，让 leader 一眼看全。
- 本期确实没有风险时，写"本期无重大风险"，不要编。
- 区分内/外因：外部资源约束（如集群维护）要注明是外因，不是任务本身的问题。

## 可读性：写给"技术型 leader"看

受众是**懂技术的 leader**——看结论、机制和影响，但不关心代码细节。用**精准的技术书面语**，
不要口语化/戏剧化。

- **专业/技术名词保留**：命中率、二级缓存、记忆投毒、benchmark、VLM、调度、provenance 等该用就用。
- **不要出现代码级字段/变量/函数/文件名、commit 哈希**（如 `_pointer_key`、`recall_failed`、
  `nodes.py`）——换成"它解决了什么问题、机制是什么、有什么影响"。
- **专业、克制，不口语/不煽情**：不要"自以为命中、省了钱""悄悄浪费""踩坑""搞定"这类表达；
  用机制化的技术描述。
  - 口语（反例）：「版本升级后缓存失效却被误判为有效，系统'自以为命中、省了钱'，成本被悄悄浪费」
  - 技术（正确）：「缓存键未绑定处理逻辑版本，版本升级后命中过期缓存、返回陈旧结果，且不触发重算，
    形成静默的缓存失效与冗余的大模型调用开销」
  - 用词上避免「首夜」「搞定」「踩坑」等，用「上线首日/初期」「完成」「问题」这类书面表达。
- 指标/数字**保留并给参照系**：如「缓存命中率约 N%，仍有约六成请求需重算，降本空间较大」。
  **给本人/leader 的实际报告一律用真实数字**；只有会公开/外传的内容（如仓库 few-shot 示例）才用占位符脱敏。
- **附链接前必须确认它对得上这条**：采集器 `links` 字段里的 URL 只是"这条会话里出现过"，
  **不等于就是这件事的产物**。只有当证据明确表明该链接就是这条的对应产物时才附——
  例如会话里明说"MR#xxx 已创建"且正是这件事的 MR；或链接标题/内容与本条主题一致。
  - **不能仅因同一会话出现过就贴**。比如某方案的会话里也出现了另一篇无关的设计文档链接，
    但那不是这件事的产物，就不能贴到该条目上。
  - 私有链接（Notion/内网 MR 等）无法打开核实时，只在有强文本证据指向它时才附，否则**宁可不附**。
  - 形如：`相关：[MR#xxx](https://…)`。不要附会话内部路径或本地文件路径。

## 通用写法规则

- **先结论后过程**：每条先给结果/状态，再补背景。具体优于笼统。
- **不杜撰**：每条都要能追溯到采集到的会话证据（user_prompts / assistant_snippets
  / files_touched / tool_activity）。证据没有的不写。
- **保留英文，不要自译**：原文是英文的术语、产品名、模型名、专有名词（如 benchmark、VLM、hook、
  Edit Plan、provenance、DeepSeek-V4-Pro 等）一律保留英文原样，不要翻成中文。中英混排没关系。
- **不臆测、不戏剧化吹捧**：不要写"怀疑/可能与 X 有关"这类没有证据的猜测；不要"最大短板/关键抓手/
  里程碑"这类自我拔高的修辞。只陈述证据支持的事实与结论。
- **下一步要对照证据核实，不要提已经做过的事**：写"下一步"前先看证据里这件事是不是已经做了。
  例如：命中率数据本身就来自现有监控（Prometheus 等），那"搭监控面板"就不是下一步（监控已存在）；
  误把已完成/已存在的当成待办，会显得不了解现状。
- **用量级信号判断篇幅与主次，但不要写进正文**：编辑次数 / 涉及文件数（`tool_activity`、
  `files_touched`）只用来**内部判断**这件事有多重、该放主线还是支线、该写多详细——
  **不要把"近 200 处编辑""涉及十余文件""基本等于重做""基于线上 MR 起步"这类元信息写进报告**。
  leader 看的是这件事本身的价值与结论，不是你改了多少行。
- **按重要度排序**：影响大、决策性、风险类的排前面；琐碎配置、一次性查询等噪音
  合并或丢弃。
- **反枚举**：一条是一件真实的事，不是关键词堆叠（连接子句 ≤ 3 个）。
- **语言**：跟随用户的语言（默认中文）。
- **不暴露内部标识**：默认不写 session_id / 完整文件路径；需要佐证工作量时用
  轻量计量，如 `(3 个会话, 改动 12 个文件)`。
