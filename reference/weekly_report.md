# 周报 Prompt

先读 `report_principles.md`（三段式、结论>数量、覆盖工作内容/进度/风险）。周报要求
与日报相同，区别在于**跨天聚合**。

## 输入（两种来源，二选一）

**A. 会话采集结果**（默认）：整周（周一至周日）的 `days` 按日期分组的 session 桶。

**B. 用户提供的日报**（"发日报→生成周报"）：用户直接把本周的若干份日报（文本/粘贴/文件）发来，
要求据此生成周报。此时**以用户的日报为唯一事实来源**，不再去翻会话记录（除非用户要求补充）：
- **若是多人共享的团队日报，先做「人物提取」**（见 `report_principles.md`）：只保留本人（或用户指定者）
  名字开头的条目，丢掉其他同事的，别误抓相邻行。
- 解析每份日报里的条目（项目、产出/结论、下一步、风险）。
- 跨天合并同一项目/主题为一条进展弧线，抽象层级高于单日。
- 风险按各日报的风险汇总、去重，合进周报「风险同步」。
- 不要新增日报里没有的事实；日报缺失的结论在周报里仍标为缺口。

## 处理
1. **跨天合并同一线程**：同一个项目/文件/任务/主题在多天推进时，合并成**一条**，
   用进展动词描述整段弧线（"从…推进到…""逐步完成…"），不要按天拆成多条。
   匹配看：项目/计划的规范名、同一文件/模块、同一反复出现的主题——不只看 ID 全等。
2. 每条仍按三段式：做了什么（整周累计）→ 产出/结论 → 下一步。
3. **抽象层级高于单日**：周报描述比任何一天都更概括。
4. **保留多样性**：进展、决策、风险、协作/外部信息都要体现，不要全压成"进展"。
5. 按重要度排序（影响/决策性/风险优先），再按覆盖广度（涉及天数/会话数）。
6. 若整周只有一天有活动：当作"本周开局快照"，少而精、抽象写。

## 输出结构（Markdown，给人读）

措辞专业、保留英文术语、不出现代码字段、不用 emoji（见 `report_principles.md`）。
分**公司周报**和**个人周报**两份**独立**文档，**每份都含完整的 5 个板块**
（总结 / 本周完成 / 进行中 / 下周重点 / 风险同步全部公司归公司、个人归个人，不要全局共享）。
「本周完成」按主题分节、按重要性排序，每节密实、带指标与链接状态；有对比数据用小表格。
下面是**单份**的模板（公司、个人各套用一次）：

```markdown
# 周报 · {week_start} ~ {week_end}

## 总结
1–2 句主线：本周核心进展/结论。

## 本周完成
### \<主题/项目（保留英文术语）\>
一段密实文字（约 2–4 句）：做了什么 + 关键结论/指标。**不在这里写下一步**。
相关：[MR#xxx 一句话说明](…) — 已合并 ✅

### \<下一个主题\>
（有评测/对比时用表格）

| 模型 | 指标 |
|---|---|
| … | … |

## 进行中
- \<正在推进、尚未完成的事\>（所有"下一步"在这里和「下周重点」汇总）

## 下周重点
- \<下周关键事项，按优先级\>

## 风险同步
- **\<风险点\>**（进度/延期/依赖/待确认/质量/成本/安全）：什么有风险 + 影响 + 应对。
- …（无则"本期无重大风险"）
```

「本周完成」按重要性排序（主项在前，可在标题后标注（主线）/（调研）/（支线）作提示）。
同一项目跨多天合并成一条进展弧线（progression）。风险既在对应条目里点到，也在末尾「风险同步」汇总。

## 出完报告后：主动询问归类

报告给出后，**主动用 AskUserQuestion 问用户**确认公司/个人归属是否正确。可先按
`project_categories.md` 给预判。拿到回答后据需要重排，并把归属写回 `project_categories.md` 与记忆。

## 自检
- [ ] 总结 1 句、不复述下面各条？
- [ ] 同一线程跨天合并成一条，没有按天重复？
- [ ] 每条都有"产出/结论"而非数量堆叠？每条约 3 句？
- [ ] 公司/个人各一份独立报告，各自带全套板块（含进行中/下周重点/风险同步）？
- [ ] 按重要度排序、不杜撰、不暴露代码字段/session_id/路径？链接确属本条产物？
- [ ] 整周无活动则说明并返回空？

## 范例（公司周报 few-shot；通用脱敏样例，仅示范结构与文风）

> 注意篇幅：每条 1–3 句即可。数字用占位符（N%/XX 万/N 条）示意"此处该给指标"，实际写真实数。

```markdown
# 周报 · 2026-XX-XX ~ 2026-XX-XX

## 总结
本周以<核心主项>从设计推进到上线与实测为主线，并行完成一项<方案设计>、一处<数据/线上修复>，并做了相关调研。
（占位符 <…> / N% 仅示意结构，实际写真实项目与真实数字。）

## 本周完成

### 1. <核心主项>——从设计到上线再到实测（主线）
背景：<为什么做，一句>。本周完成设计与实现（<关键设计点 1、2>），修复一处<某类> bug（导致<什么后果>的真实问题），
N 条测试通过并上线；上线后实测<关键指标> X%→Y%（节省/提升约 Z%），仍约 N 成<未覆盖>。
相关：[MR#xxx](…)
下一步：补<累计硬指标>；评估<存量/回填>。
（说明：同一主项的"实现/上线"与"指标实测"是两个阶段，合并为这一条；此为「正常版」——简洁版不在条目内写下一步、统一收进进行中/下周重点。）

### 2. <数据/线上修复>（支线）
排查<某条链路/字段>，明确<边界/口径>，最小改动修复并提 MR。
相关：[MR#xxx](…)

### 3. <方案设计>（调研/设计）
评估现状后产出一版可落地、高 ROI 的方案，并明确优先级（<A > B > C>）。

### 4. <调研/梳理>（调研）
通读<某链路>形成完整认知，并调研相关资料梳理改进方向。

## 进行中
- <主项> MR 评审，及指标量化补充。
- <方案> 待与相关团队对齐执行边界。
- 调研方向收敛为可执行项。

## 下周重点
- 推动 <主项> MR 合入，补全硬指标。
- <方案> 对齐边界、落地。
- 调研收敛为可执行项。

## 风险同步
- **依赖**：<主项>与<修复> MR 均待评审合入，未合入前收益无法计入线上。
- **质量**：<指标> 约 N%、未达理想；累计硬数字待算。
- **跨团队待确认**：<某执行项>边界需对齐，存在返工可能。
```
