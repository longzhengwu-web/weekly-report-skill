# 周报 Prompt

先读 `report_principles.md`（三段式、结论>数量、覆盖工作内容/进度/风险）。周报要求
与日报相同，区别在于**跨天聚合**。

## 输入
整周（周一至周日）的会话采集结果：`days` 下按日期分组的 session 桶。也可以先逐日按
`daily_report.md` 理解每天，再聚合成周。

## 处理
1. **跨天合并同一线程**：同一个项目/文件/任务/主题在多天推进时，合并成**一条**，
   用进展动词描述整段弧线（"从…推进到…""逐步完成…"），不要按天拆成多条。
   匹配看：项目/计划的规范名、同一文件/模块、同一反复出现的主题——不只看 ID 全等。
2. 每条仍按三段式：做了什么（整周累计）→ 产出/结论 → 下一步。
3. **抽象层级高于单日**：周报描述比任何一天都更概括。
4. **保留多样性**：进展、决策、风险、协作/外部信息都要体现，不要全压成"进展"。
5. 按重要度排序（影响/决策性/风险优先），再按覆盖广度（涉及天数/会话数）。
6. 若整周只有一天有活动：当作"本周开局快照"，少而精、抽象写。

## 输出结构（Markdown，给人读）

先产出**一份统一周报**（不预分公司/个人），措辞专业、保留英文术语、不出现代码字段
（见 `report_principles.md`）。**不要"本周概览"**，**不要 emoji/图标**。
每条三段式、先给产出/结论，结论要带决策思路与量化影响，下一步要具体带优先级。

**周报按"工作性质"分三层**（让 leader 一眼看出主次），每层内再按重要性排序：

- **主线任务** —— 本周的核心工作、你的主项、对目标贡献最直接的产出（最重要，放最前）。
- **调研 / 设计类** —— 调研、方案设计、流程梳理、评测标准这类"想清楚/打基础"的工作（有产出但尚未交付）。
- **支线 / 协作类** —— 需要做、但不是自己主项的事：支撑性修复、跨团队配合、维护类任务。

```markdown
# 周报 · {week_start} ~ {week_end}

## 主线任务
### \<项目/主题（通俗名字，保留英文术语）\>
- **做了什么**：本周从…推进到…
- **产出/结论**：指标/数字（带参照系）+ 决策思路 + 解决了什么问题。
- **下一步**：具体做什么、什么节奏、为什么重要。
  相关：[MR/Notion/文档链接](…)

## 调研 / 设计类
### \<主题\>
- …

## 支线 / 协作类
### \<主题\>
- …

## 风险同步
- **\<风险点\>**（类型：进度/延期/依赖/待确认/质量/成本/安全）：什么有风险 + 原因 + 影响 + 应对或所需支持。
- …（本期无重大风险则写"本期无重大风险"）
```

分层判定：按工作**性质**归层（交付主项→主线；调研/设计/梳理→调研；支撑/配合/维护→支线），
某层无内容则省略整层。同一项目跨多天合并成一条进展弧线（progression）。
风险既在对应条目里点到，也在末尾「风险同步」汇总（见 `report_principles.md`）。

## 出完报告后：主动询问归类

报告给出后，**主动用 AskUserQuestion 问用户**：(1) 哪些属于"公司"、哪些属于"个人"；
(2) 主线/调研/支线的分层对不对。可先按 `project_categories.md` 给预判。拿到回答后据需要重排，
并把归属/分层写回 `project_categories.md` 与记忆，供下次预判。

## 自检
- [ ] 同一线程跨天合并成一条（含进展动词），没有按天重复？
- [ ] 每条都有"产出/结论"而非数量堆叠？
- [ ] 进度、风险、决策、协作多个面向都覆盖，没全压成进展？
- [ ] 抽象层级高于单日流水？
- [ ] 按重要度排序、反枚举、不杜撰、不暴露 session_id/路径？
- [ ] 整周无活动则说明并返回空？
